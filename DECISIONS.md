# Ключевые архитектурные решения

## 1. Без фреймворка — собственный микрокерас

**Решение:** Реализован минимальный фронт-контроллер (`Router`, `App`) без внешних HTTP-фреймворков (Laravel, Symfony, Slim).

**Причина:** Задача учебная, зависимости минимальны. Весь роутинг — 60 строк кода. Добавление фреймворка утяжелило бы проект без реальной пользы.

---

## 2. SQLite + WAL-режим как гарантия атомарности

**Решение:** `PRAGMA journal_mode=WAL` + транзакция + условный `UPDATE WHERE status = 'assigned'`.

**Причина:** SQLite сериализует записи: только один из двух конкурентных `UPDATE` изменит строку. `rowCount()` сразу показывает победителя без дополнительного `SELECT`. WAL-режим позволяет одновременные чтения, не теряя надёжности.

---

## 3. session_write_close() перед критической секцией

**Решение:** В `MasterController::take()` вызывается `session_write_close()` до обращения к БД, затем `session_start()` для записи flash-сообщения.

**Причина:** PHP по умолчанию держит файловую блокировку сессии на всё время запроса. Два параллельных запроса с одной сессией были бы сериализованы на уровне PHP, а не БД — и тест гонки не имел бы смысла. Ранний `session_write_close()` освобождает lock и позволяет истинному параллелизму дойти до БД.

---

## 4. Слоистая архитектура: Controller → Service → Repository

**Решение:** Явное разделение на три слоя: Controllers (HTTP), Services (бизнес-логика), Repositories (доступ к данным).

**Причина:** Сервисы можно тестировать без HTTP-стека. Репозитории можно заменить без изменения логики. Controllers тонкие — только валидация входа и редирект.

---

## 5. Audit log как отдельная таблица

**Решение:** Каждое изменение статуса пишется в `audit_log` (кто, когда, что изменил, старый/новый статус).

**Причина:** Требование ТЗ. Отдельная таблица не замедляет чтение заявок и позволяет строить историю без изменения структуры `service_requests`.

---

## 6. Авторизация — сессия + password_hash

**Решение:** Простой логин по username/password, хранение `user_id` и `role` в `$_SESSION`. Пароли — `password_hash(PASSWORD_BCRYPT)`.

**Причина:** ТЗ допускает упрощённую авторизацию. Bcrypt — минимальный необходимый стандарт безопасности даже для учебного проекта.

---

## 7. In-memory SQLite для тестов

**Решение:** `Database::configure(':memory:')` в `tests/bootstrap.php`. Синглтон `Database` поддерживает `reset()` для изоляции интеграционных тестов.

**Причина:** Тесты работают без файловой системы, быстро, без очистки после себя. `reset()` позволяет создавать изолированные файловые БД для race condition тестов.
