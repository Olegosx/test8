## 2. Архитектурное решение по race condition

При проектировании `takeInProgress` был проработан механизм защиты:

- Вариант A: `SELECT FOR UPDATE` — недоступен в SQLite.
- Вариант B: Оптимистичная блокировка + версионирование — лишняя сложность.
- **Вариант C (выбран):** Условный `UPDATE WHERE status = 'assigned'` в транзакции.
  SQLite сериализует записи в WAL-режиме; `rowCount()` сразу даёт результат.

Дополнительно обнаружена проблема PHP-сессионного лока: решена через
`session_write_close()` перед обращением к БД в `MasterController::take()`.

## 3. Структура тестового покрытия

При написании тестов были охвачены:

- Happy path: создание, назначение, взятие в работу, завершение.
- Edge cases: нельзя взять `new`-заявку, нельзя отменить `done`, нельзя
  завершить `assigned`, нельзя взять чужую заявку.
- Race condition: `testSecondTakeReturnsFalse` + интеграционный тест с двумя
  PDO-соединениями к одному SQLite-файлу.
- Audit trail: проверка полноты истории действий.

## 4. Шаблонизация без фреймворка

Использован паттерн `ob_start() / ob_get_clean()` в каждом шаблоне
с последующим `include layout.php`. Позволяет переиспользовать разметку
без движка шаблонов (Twig и т.п.).
